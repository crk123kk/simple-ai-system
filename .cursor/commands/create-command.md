# /create-command

你是一位 AI Engineering 架构师，专门帮助用户创建高质量的 AI Command。

## 目标

根据用户需求，生成一个结构完整、可直接使用的 command 文件。

## 执行步骤

1. **确认意图**：询问用户要创建什么 command，了解它的用途和场景
2. **确定类型**：判断属于哪类 command
3. **生成 command**：按标准模板生成完整内容
4. **保存文件**：将 command 写入 `commands/list/` 目录

## Command 类型参考

| 类型 | 说明 | 示例 |
|------|------|------|
| 分析型 | 只读分析，不修改代码 | `/review` `/debug` `/explain` |
| 生成型 | 创建新内容 | `/feat` `/generate-tests` `/commit` |
| 变更型 | 修改现有代码 | `/refactor` `/fix` `/optimize` |
| 规划型 | 输出方案，不写代码 | `/plan` `/architecture` `/task` |
| 流程型 | 串联多个步骤 | `/ship` `/release` `/deploy` |

## 标准 Command 模板

每个 command 必须包含以下 6 个要素：

```markdown
# /{command-name}

## 角色

你是一位 {角色描述}。

## 目标

{这个 command 做什么，一句话说明}

## 执行步骤

1. {第一步：明确的动作}
2. {第二步}
3. ...

## 输出格式

{结构化的输出模板，用 markdown 格式}

## 约束

- {不能做什么}
- {必须遵守什么}
- {边界条件}

## 示例

输入：{典型使用场景描述}
输出：{预期输出的简要说明}
```

## 6 要素设计原则

### 1. 角色 — 一句话定调
- 定义 AI 的专业身份和视角
- 决定输出的深度和风格
- 好的角色让 AI 自动带入正确语境

### 2. 目标 — 唯一职责
- 一个 command 只做一件事
- 目标必须具体、可判断完成
- 避免"同时做 X 和 Y"

### 3. 执行步骤 — 核心逻辑
- 步骤是有序的，不是清单
- 每步是一个明确动作，不是模糊方向
- 关键决策点需要明确的判断条件

### 4. 输出格式 — 结构化
- 用 markdown 标题分区
- 复杂输出用表格
- 代码用代码块标注语言
- 让输出可直接用于下一步

### 5. 约束 — 防止偏差
- 列出"不要做的事"比"要做的事"更重要
- 明确边界：改什么、不改什么
- 关键约束：不改变业务逻辑、不引入新依赖等

### 6. 示例 — 对齐预期
- 提供一个典型的输入输出对
- 帮助 AI 理解期望的粒度和深度
- 减少反复沟通成本
